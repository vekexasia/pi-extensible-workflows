Developer path

Build and operate workflows

This page describes the package contract from the host tool boundary through the sandboxed workflow worker and its durable run store.

Installation and trust

Use Node.js 22.19 or newer. pi-extensible-workflows is a trusted Pi extension: installing it gives the extension the same filesystem and process access as Pi. Only install it from a source you trust, and open and trust the project in Pi before running doctor.

npm ci
npm run check
pi install /absolute/path/to/pi-extensible-workflows

For a one-session trial without changing Pi settings:

pi --no-extensions --extension /absolute/path/to/pi-extensible-workflows/src/index.ts

The package declares Pi AI, Pi coding agent, and TypeBox as peer dependencies. The repository development setup currently pins the Pi packages to 0.80.9.

Tool API

The extension registers two tools and one command:

  • workflow runs an inline or registered deterministic JavaScript workflow.
  • workflow_respond approves or rejects a pending checkpoint.
  • /workflow lists and controls runs in the current Pi session.

workflow parameters

FieldTypeContract
namestringRequired for an inline script; registered workflows use their registered name.
descriptionstringOptional human-readable metadata.
scriptstringImmutable workflow source. Mutually exclusive with workflow.
workflowstringRegistered reusable workflow as namespace.name.
argsJSON valueAvailable inside the script as args; defaults to null.
foregroundbooleanWait for the final value instead of returning a run ID.
concurrencyinteger 1-16Per-run active-agent limit.
maxAgentLaunchespositive integerTotal logical agent launches, including nested children.

workflow_respond parameters

{
  "runId": "...",
  "name": "ship",
  "approved": true
}

Background workflow calls return a run ID immediately. Completion or failure is delivered as one follow-up. Foreground calls return the workflow value inline.

Workflow DSL

Workflow source is plain sandboxed JavaScript. It has no imports, filesystem, network, process, timer, or dynamic-code globals. The deterministic primitives are:

PrimitiveUse
agent(prompt, options)Run one Pi agent and return its bare text or JSON value.
prompt(template, values)Safely interpolate JSON-compatible values into a prompt.
parallel(name, tasks)Run keyed independent tasks concurrently.
pipeline(name, items, stages)Apply ordered stages to keyed items.
phase(name)Record progress telemetry.
log(message)Add a TUI-only transcript entry, capped at 4 KB.
checkpoint(input)Wait for an approval or rejection.
withWorktree(callback)Create one shared worktree lazily for a collaborating scope.

Agent options

Use model as provider/model, optionally with a thinking suffix such as :high. Other options are thinking, role, tools, outputSchema, retries, timeoutMs, and isolation: "worktree". A role owns its model, thinking, and tools policy, so do not combine role with those three overrides.

const report = await agent("Review the implementation", {
  model: "openai-codex/gpt-5.6-sol:high",
  tools: ["read", "grep"],
});

Results are bare values. With outputSchema, the native agent submits the schema through workflow_result; prose is not parsed as structured output. The serialized result and every worker RPC value must fit within 10 MB.

Composition

const reports = await parallel("review", {
  api: () => agent("Inspect the API"),
  ui: () => agent("Inspect the UI"),
});
return agent(prompt("Combine these reports:\n{reports}", { reports }));

Calls from one source location must not race outside parallel or pipeline. Their structural keys make replay deterministic. Ordinary branch failures wait for sibling work to settle; cancellation is immediate.

Extensions, reusable workflows, and roles

Trusted Pi extensions can register global functions, run-scoped variables, and reusable workflows while extensions load. Registration validates the namespace, semantic version, descriptions, schemas, and workflow source.

import { registerWorkflowExtension } from "pi-extensible-workflows";
registerWorkflowExtension({
  namespace: "review",
  version: "1.0.0",
  headline: "Review helpers",
  description: "Reusable review primitives",
  functions: {
    review: {
      description: "Review one focus",
      input: { type: "object" },
      output: { type: "string" },
      async run(input, { agent }) {
        return agent(`Review ${String(input.focus)}`);
      },
    },
  },
  workflows: {
    release: {
      description: "Review release readiness",
      script: `return review({ focus: "release readiness" });`,
    },
  },
});

Call the registered workflow as { "workflow": "review.release" }. Registered scripts call functions and read variables directly. When available, workflow_catalog exposes flat metadata without implementations, resolvers, values, or source.

Role files

Global roles live under ~/.pi/pi-extensible-workflows/roles/<name>.md. Trusted project roles live under <cwd>/.pi/pi-extensible-workflows/roles/<name>.md. The legacy ~/.pi/piworkflows/roles/ and <cwd>/.pi/piworkflows/roles/ paths remain readable for compatibility, with the new paths taking precedence when both define a role.

---
description: Reviews code for correctness
model: anthropic/claude-fable-5
thinking: high
tools: [read, grep]
---
Focus on correctness and regressions.

When role is present, omit model, thinking, and tools from the agent call. The role body is appended to the native Pi system prompt.

Lifecycle, checkpoints, and isolation

Run states are queued, running, pausing, paused, awaiting_input, completed, failed, stopped, and interrupted. Agent states include queued, running, waiting_for_child, paused, retrying, completed, failed, and cancelled.

Use a checkpoint when an agent must wait for a human or parent decision:

const decision = await checkpoint({
  name: "ship",
  prompt: "Ship this build?",
  context: { commit: args.commit },
});
if (decision !== "approved") return { approved: false };

Checkpoint prompts are limited to 1 KB and serialized context to 4 KB. Direct checkpoint() returns "approved" or "rejected". Responses are journaled and replay after cold recovery.

Top-level agents can request isolation: "worktree". The runtime creates a deterministic owned branch and worktree from the launch tree. The caller branch is unchanged; children and retries reuse the worktree. Use withWorktree(name, callback) when collaborating agents must share one worktree.

Runs are stored under ~/.pi/workflows/projects/<cwd-slug>-<cwd-hash>/sessions/<session-id>/runs/<run-id>/. Identity checks use the resolved launch cwd and Pi session ID.

Operations and debugging

Doctor and inspector

npx pi-extensible-workflows doctor
npx pi-extensible-workflows inspect [session-id]
npm run inspect -- [session-id]

doctor is read-only and checks Pi trust and resources, active tools, models, roles, settings, extension load failures, and registered workflows. It exits nonzero only for errors. inspect opens the read-only terminal inspector; the detail view includes agents, retries, models, costs, prompts, and the syntax-highlighted workflow script.

/workflow controls

/workflow doctor
/workflow pause <run-id>
/workflow resume <run-id>
/workflow stop <run-id>
/workflow approve <run-id> <checkpoint>
/workflow reject <run-id> <checkpoint>
/workflow delete <run-id>

The navigator discovers only runs from the exact current cwd and Pi session. Pause is cooperative, stop is immediate and irreversible, and delete is available only for terminal runs after confirmation.

Settings and errors

{
  "concurrency": 8,
  "maxAgentLaunches": 1000
}

Optional strict settings live at ~/.pi/workflows/settings.json. Invocation options override the defaults. The session-wide active-agent ceiling is 16.

Host-facing failures are WorkflowError values with stable codes such as INVALID_SYNTAX, UNKNOWN_MODEL, UNKNOWN_TOOL, AGENT_FAILED, AGENT_TIMEOUT, RESULT_INVALID, CANCELLED, WORKTREE_FAILED, and RESUME_INCOMPATIBLE. Fix the reported trust, model, tool, role, or script issue before retrying.

Evaluation and package checks

Deterministic evaluation helpers use a fake bridge and make zero model calls during normal tests. Model-backed evaluations are opt-in and require an explicit provider and model.

npm run check
npm run acceptance
npm run evals -- --provider "$PROVIDER" --model "$MODEL" --case direct-answer,parallel
PI_WORKFLOW_EVAL_AMBIENT=1 npm run evals:ambient -- --provider "$PROVIDER" --model "$MODEL"

The ambient tier is separately opt-in. Do not enable it in ordinary CI unless its external model cost and cleanup are intentional.
